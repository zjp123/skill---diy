# CI 门禁 & 质量规则

所有阶段共用的约束规则。PR 合并和发布前必须通过所有相关 Gate。

> 所有命令均使用项目已有的包管理器和脚本（从 Phase 2 确认），不预设 `npm` / `pnpm` / `yarn` / `bun`。

---

## PR 合并前门禁（自动化）

```
Gate-1  REQ 覆盖率 = 100%
        每个 REQ 至少映射一个任务与一个验收项

Gate-2  Lint 通过
        使用项目已有 lint 命令

Gate-3  类型检查通过
        使用项目已有 typecheck 命令

Gate-4  核心单测通过
        覆盖范围：输入校验 / 权限渲染 / 核心业务逻辑 / 状态机（根据 Phase 1 状态机结论）

Gate-5  主链路 E2E 通过
        覆盖 Phase 8 中确认的主链路场景（根据本次 PRD 动态确定）

Gate-6  高风险项人工抽检通过
        根据 Phase 0 安全约束和 Phase 3 权限态设计，确认权限边界 / 鉴权流程 / 数值精度 / 状态机边界
```

---

## 发布前额外门禁

```
Gate-7  回归清单通过
        列出本次改动影响的关联系统范围（根据项目实际下游依赖确定）

Gate-8  灰度观测通过
        核心操作成功率 / 错误率在项目约定阈值内

Gate-9  回滚开关验证通过
        灰度失败可一键回滚
```

---

## 全局禁止事项

| 禁止 | 原因 |
|---|---|
| 凭空假设第三方库已存在 | 必须先在 `package.json` 验证 |
| 泄露敏感信息或密钥 | 安全红线 |
| 测试中使用固定时间等待 | 导致不稳定测试 |
| 用 `any` / 类型断言规避类型问题 | 掩盖真实类型错误 |
| 修改 lint / 类型检查 / 构建配置来通过检查 | 只允许修复代码本身 |
| 创建无关文档 | 优先在已有交付链路内补充 |

---

## 可追溯性要求

所有 PR 必须满足以下追溯链：

```
代码  → REQ    通过 PR 描述 Implements: REQ-*
测试  → REQ    通过 PR 描述 Tests: case-*
文件  → REQ    通过骨架映射表
PR    → 风险   通过 PR 描述 Risk: high/medium/low
```

---

## 门禁失败处理机制

| 情形 | 处理 |
|---|---|
| 任一门禁失败 | 任务状态从 Done 回退到 In Progress |
| REQ 映射缺失 | 禁止进入测试阶段 |
| 回归失败 | 仅允许修复回归问题，不允许顺带改需求范围 |
| CRITICAL 安全问题 | 必须修复后重新走完整评审，不允许 fast-track |
