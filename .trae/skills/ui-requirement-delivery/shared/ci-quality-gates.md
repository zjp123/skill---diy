# CI 门禁 & 质量规则

所有阶段共用的约束规则。PR 合并和发布前必须通过所有相关 Gate。

---

## PR 合并前门禁（自动化）

```
Gate-1  REQ 覆盖率 = 100%
        每个 REQ 至少映射一个任务与一个验收项

Gate-2  lint 通过
        npm run lint

Gate-3  typecheck 通过
        npm run typecheck 或 tsc --noEmit

Gate-4  核心单测通过
        输入校验 / 权限渲染 / 汇总逻辑 / 状态机

Gate-5  主链路 E2E 通过
        保存配置 → 数据展示一致

Gate-6  高风险项人工抽检通过
        权限越权 / 2FA 流程 / 金额精度 / 状态机边界
```

---

## 发布前额外门禁

```
Gate-7  回归清单通过
        跨系统影响范围（列出本项目关联的下游系统，如：Admin / Portal / BFF 等）

Gate-8  灰度品牌观测通过
        错误率 / 保存成功率在阈值内

Gate-9  回滚开关验证通过
        灰度失败可一键回滚
```

---

## 全局禁止事项

| 禁止 | 原因 |
|---|---|
| 凭空假设第三方库已存在 | 必须先在 `package.json` 验证 |
| 泄露敏感信息或密钥 | 安全红线 |
| 测试中使用固定 `waitForTimeout` | 导致不稳定测试 |
| 用 `any` 规避类型问题 | 掩盖真实类型错误 |
| 修改 lint / tsconfig 配置来通过检查 | 只允许修复代码本身 |
| 创建无关文档 | 优先在已有交付链路内补充 |

---

## 可追溯性要求

所有 PR 必须满足以下追溯链：

```
代码  → REQ    通过 PR 描述 Implements: REQ-*
测试  → REQ    通过 PR 描述 Tests: case-*
文件  → REQ    通过骨架映射表
PR    → 风险   通过 PR 描述 Risk: high/medium/low
```

---

## 门禁失败处理机制

| 情形 | 处理 |
|---|---|
| 任一门禁失败 | 任务状态从 Done 回退到 In Progress |
| REQ 映射缺失 | 禁止进入测试阶段 |
| 回归失败 | 仅允许修复回归问题，不允许顺带改需求范围 |
| CRITICAL 安全问题 | 必须修复后重新走完整评审，不允许 fast-track |
