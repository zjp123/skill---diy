# User Confirmation Protocol — 用户确认协议

## 核心原则

> **AI 遇到不确定时，必须先问，不得自行假设后继续执行。**
> 用户确认完成之前，当前 Phase 的后续步骤全部冻结。

这不是建议，是强制约束。任何违反本协议的"擅自推进"行为均视为工作流执行错误。

---

## 必须暂停询问的场景（MUST STOP）

以下场景出现时，AI **立即停止当前 Phase**，向用户提出确认问题，等待回复后才能继续：

### S1 — 需求歧义

| 触发条件 | 示例 |
|---|---|
| PRD 描述与 Figma 设计不一致 | PRD 说"不分页"，Figma 有分页控件 |
| 同一功能在 PRD 中有多种理解 | "支持多选"是全局多选还是单行多选 |
| 前端 / 后端职责边界不明确 | 数据聚合逻辑在前端算还是接口返回 |

### S2 — 优先级 / 范围不明

| 触发条件 | 示例 |
|---|---|
| 无法判断功能是 P0 还是 P1 | 首发是否必须包含「导出」功能 |
| Sprint 排期冲突 | FE-05 依赖 FE-03，但 FE-03 在 Sprint 2，FE-05 想进 Sprint 1 |
| 需求超出 Figma 设计范围 | PRD 提到权限管控但 Figma 无对应 UI |

### S3 — 组件 / 架构选型有重大分歧

| 触发条件 | 示例 |
|---|---|
| 有 2 个以上可复用组件，差异明显 | 两个 Modal 组件 API 不同，选哪个 |
| 需要引入新依赖才能满足需求 | 现有组件不支持虚拟滚动，要不要引入虚拟滚动库 |
| 全局状态 vs 局部状态选择影响多个文件 | 某状态是放全局 Store 还是页面级局部状态 |

### S4 — 接口文档与项目规范冲突

| 触发条件 | 示例 |
|---|---|
| 后端字段命名与项目前端规范不一致 | 后端返回 `user_name`，前端统一用 `userName` |
| 接口返回结构与项目已有包装层不匹配 | 新接口裸返回数组，旧接口都有 `{ data, code }` |
| 错误码在文档中未定义 | 后端文档没有 `409` 的处理说明 |

### S5 — 安全 / 权限相关决策

| 触发条件 | 示例 |
|---|---|
| 不确定该操作是否需要 2FA | 修改汇率是否需要二次验证 |
| 权限粒度不明确 | 「查看」和「编辑」权限是同一个 key 还是分开 |
| 路由是否需要鉴权保护 | 新页面是公开访问还是登录后才可见 |

### S6 — 发现已有代码与 PRD / Figma 矛盾

| 触发条件 | 示例 |
|---|---|
| 侦察发现现有实现逻辑与 PRD 要求不同 | 现有代码是单选，PRD 要求多选 |
| 发现可能影响现有功能的改动 | 修改共享 Hook 可能影响其他页面 |

## 提问格式（必须遵守）

每次暂停只问**一个核心问题**，不堆叠多个疑问。若有多个疑问，按优先级逐一提问。

```
⚠️ 需要您确认后才能继续

**场景**：[一句话说明遇到了什么不确定]

**我的理解**：[AI 当前的理解或倾向]

**问题**：[一个具体、可直接回答的问题]

**选项**（可选，有明确选项时提供）：
- A. [选项 A 及其影响]
- B. [选项 B 及其影响]

> 收到您的回复后，我将继续执行 [Phase X 的下一步骤]。
```

### 示例

```
⚠️ 需要您确认后才能继续

**场景**：Phase 0 解析 PRD 时，「操作历史记录」功能在文档中标注为「首期上线」，
但项目经理 @xxx 的备注中写「可延后」，两处说法矛盾。

**我的理解**：倾向于按文档正文执行，标注为 P0。

**问题**：「操作历史记录」是否进入首期上线（P0）？

- A. 是，P0，进入 Sprint 1（按文档正文）
- B. 否，P1，延后到 Sprint 2（按备注说明）

> 收到您的回复后，我将完成 REQ 主键表并输出 prd-analyze.md。
```

---

## 等待期间的约束

用户确认前，AI **禁止**：

- 继续执行当前 Phase 的后续步骤
- 跳过当前 Phase 进入下一 Phase
- 用「我假设 X，继续执行」代替真正的暂停
- 在同一条消息里提问然后自己回答

---

## 可以不暂停的情况（SAFE TO PROCEED）

以下情况 AI 可以自行决策，但必须在输出中**以注释形式标注假设**：

| 情况 | 处理方式 |
|---|---|
| 命名风格已有明确的项目规范 | 按规范执行，注释「遵循项目 camelCase 规范」 |
| 组件选型只有一个合理选项 | 直接使用，注释「仅有 X 符合要求，直接复用」 |
| 边界值处理有行业惯例 | 按惯例处理，注释「按行业惯例处理空值为 null」 |
| PRD 有明确的优先级标注且无矛盾 | 直接使用，无需额外确认 |

---

## 与 Session Log 的联动

每次暂停询问后，无论结果如何，都在 `workflow-session-log.md` 写入一条记录：

- 用户回复且继续执行 → 写入 `[CONFIRM]` 记录
- 用户要求修改方向 → 写入 `[CHANGE]` 记录
- 用户要求重新生成 → 写入 `[REGEN]` 记录

---

## 各 Phase 的高频确认触发点速查

| Phase | 最常见的暂停触发场景 |
|---|---|
| Phase 0 PRD 解析 | 功能优先级矛盾、前后端职责边界 |
| Phase 1 Figma 读取 | PRD 与 Figma 不一致、设计 Token 无法映射到项目 |
| Phase 2 代码库侦察 | 发现现有实现与需求冲突 |
| Phase 3 组件映射 | 多个组件可选且差异明显、需引入新依赖 |
| Phase 4 任务清单 | Sprint 排期冲突、任务依赖无法满足 |
| Phase 5b 接口集成 | 后端字段命名与前端规范冲突、缺失错误码定义 |
| Phase 5 代码骨架 | 文件结构有多种合理方案时 |
| Phase 6 TDD | 测试用例覆盖范围超出 PRD 描述，需确认是否纳入 |
