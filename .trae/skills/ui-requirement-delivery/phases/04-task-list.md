# Phase 4：任务清单 & RTM

## 目标
将功能点拆解为可排期、可分工、可验收的开发任务，输出 `prd-dev-task-list.md`。

---

## WBS 任务格式

每个任务必须包含：

| 字段 | 说明 |
|---|---|
| 编号 | `FE-01`, `FE-02` ... 顺序递增 |
| 任务描述 | 具体动作（动宾短语），不写空泛叙述 |
| 前置依赖 | 任务编号或外部条件（如「接口联调完成」） |
| DoD | 可量化的完成定义，评审时可直接打勾 |
| 关联 REQ | `REQ-*` 编号，**不允许空缺** |
| 埋点需求 | 若该任务涉及关键用户操作（保存 / 提交 / 权限拦截 / 报错），在 DoD 中标注埋点事件名；无需埋点则填「无」 |

> **埋点标注原则**：不需要所有任务都埋点，仅在关键操作节点标注。
> 格式示例：`DoD: <操作>成功后触发 track('<事件名>', { <关键属性> })`，事件名与属性从 Phase 2 确认的埋点规范中取。

### 示例（根据实际需求动态填写）

| 编号 | 任务 | 前置依赖 | DoD | 关联 REQ |
|---|---|---|---|---|
| FE-01 | （具体动宾任务描述） | （外部条件或前置任务） | （可量化验收条件） | REQ-001 |
| FE-02 | （具体动宾任务描述） | FE-01 | （可量化验收条件） | REQ-002 |

---

## 里程碑划分

- **M1（P0）**：主链路必须首发功能
- **M2（P1）**：体验增强与监控优化

---

## Sprint 规划建议

```
Sprint 1  基础架构（路由、权限、骨架、i18n 等基础能力）
Sprint 2  核心业务（表单、表格、数据联调、安全校验等核心功能）
Sprint 3  质量保障（单测、E2E、回归、灰度）
```

---

## RTM 映射规则

```
任务编号  →  REQ-*（多对多关系）

规则：
- 新增任务必须同步登记覆盖需求 ID
- 出现未映射 REQ 的任务，默认视为不可合并项
- RTM 表是 PR 描述中 Implements: REQ-* 的权威来源
```

### RTM 表示例

| 任务编号 | 覆盖需求 ID |
|---|---|
| FE-01 | REQ-001 |
| FE-02, FE-03 | REQ-002 |
| FE-05, FE-06 | REQ-003 |

## 联调前提条件确认（Kick-off Checklist）

> 本节只做「是否已知」的确认，详细接口实现由 **Phase 5b** 负责，不在此重复。

在进入开发之前，必须与后端对齐以下四项，若任一项「未知」则记录阻塞并通知负责人：

| 确认项 | 状态（已知 / 未知 / 待确认） | 负责人 |
|---|---|---|
| 接口字段命名（camelCase vs snake_case） | | |
| 空值语义（`null` vs `undefined` vs 空字符串） | | |
| 时间格式与时区口径 | | |
| 错误码到前端提示的映射表 | | |

> 若已提供接口文档（Swagger / OpenAPI / Postman 等），以上四项应在 **Phase 5b** 中一并输出为正式映射表，不需要在此重复填写。

## 输出检查

- [ ] 所有任务有 DoD 和 REQ 映射
- [ ] 无任何任务与 REQ 脱钩
- [ ] Sprint 规划可直接抄入迭代计划
- [ ] 联调前提条件已确认（或阻塞项已记录负责人）
- [ ] 包含埋点需求的 WBS 任务已标注（见「WBS 任务格式」中的埋点说明）
