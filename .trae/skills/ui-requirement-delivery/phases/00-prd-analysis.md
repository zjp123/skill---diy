# Phase 0：PRD 解析

## 目标
将产品文档转化为前端可执行的功能点清单。

---

## 步骤

### 1. 模块拆解

- 按页面/功能模块切分，每个模块标注 P0/P1 优先级
- 明确前端范围（哪些是前端实现、哪些是后端/服务端）

### 2. 每个功能点补充

- 接口依赖（API 契约、字段精度、时区要求）
- 异常态（空态 / 错误态 / 无权限态）
- 验收口径（可量化的 DoD）
- 安全约束（2FA / 权限 / CSRF / XSS）

### 3. 接口契约对齐（前端视角）

对每个 API 端点确认：

| 检查项 | 说明 |
|---|---|
| HTTP 方法 | GET 只读 / POST 创建 / PUT 全量替换 / PATCH 局部更新 |
| 状态码映射 | `400` 校验失败 / `401` 未登录 / `403` 无权限 / `404` 不存在 / `409` 冲突 / `422` 语义错误 / `429` 限流 |
| 分页策略 | 小数据集用 offset；列表流 / 无限滚动用 cursor |
| 错误格式 | 字段名、message 语言、details 结构统一 |
| 精度时区 | 金额精度与时区口径冻结，合并前禁止变更 |

### 4. 建立 REQ 主键表（单一事实源）

```
| 需求ID  | 需求描述           | 来源模块 | 验收关键字         |
|---------|-------------------|----------|-------------------|
| REQ-001 | 功能描述           | 模块A    | 可量化验收条件     |
| REQ-002 | ...               | ...      | ...               |
```

**使用规则**：
- 后续所有任务、UI 拆解、骨架、测试用例均引用 `REQ-*`
- PRD 变更只允许在此表增删改并记录影响范围
- 禁止在其他文档中新增平行编号体系

### 5. 输出 `prd-analyze.md`

---

## 输出检查

- [ ] 每个功能点有明确的前端范围和接口依赖
- [ ] P0/P1 优先级已标注
- [ ] 接口分页策略已确认（offset / cursor）
- [ ] 错误码 → 前端提示映射表已建立
- [ ] REQ 主键表已建立，无重复编号
