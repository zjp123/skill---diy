# Phase 3：UI → 组件映射

## 目标
将 Figma 区域拆解为可编码的组件选型，输出 `ui-analysis-with-components.md`。

---

> **确认协议**：本 Phase 遇到以下情况时必须暂停：
> 有 2 个以上组件可选且差异明显 / 需引入现有项目未使用的新依赖。
> 提问格式见 `shared/user-confirmation-protocol.md`。

## 拆解维度

每个 UI 区域必须覆盖以下所有维度：

| 维度 | 内容 |
|---|---|
| 结构 | 布局层级、嵌套关系 |
| 组件选型 | 优先复用 → 小改造 → 自定义（三选一，附代码证据） |
| 状态机 | 从 Phase 1 状态机结论中引用；若 Phase 1 标注「无状态变化」则此项填「N/A」 |
| 交互流 | 点击路径、数据流向、防抖 / 防重提交策略 |
| 校验规则 | 前端拦截规则 + 后端错误码二次兜底映射 |
| 权限态 | 无权限时的 UI 降级方案（隐藏 / 置灰 / 跳转） |
| 无障碍 | ARIA 角色、键盘导航顺序、焦点管理 |

## 状态管理选型决策

> 不要预设方案——根据 Phase 2 已确认的项目状态管理方案，对每个数据域做归属决策。

在组件映射阶段确定每个数据域的状态归属，避免后续随意混用：

| 数据类型 | 归属原则 | 判断依据 |
|---|---|---|
| 服务端数据（列表、详情、分页） | 使用项目已有的数据请求层方案 | 需要缓存、自动失效、loading/error 三态 |
| 全局 UI 状态（当前用户、权限、主题） | 使用项目已有的全局状态方案 | 多个无关组件共享，生命周期跨页面 |
| 跨步骤表单 / 向导流程 | 使用项目已有的局部共享状态方案 | 单一流程内共享，完成后销毁 |
| 单页面表单输入 | 使用项目已有的表单方案 | 需要校验、dirty 状态、提交拦截 |
| 组件本地 UI 状态（弹窗开关、编辑态） | 使用框架原生本地状态 | 不需要跨组件共享 |

**决策规则**：
- 优先用最小作用域，能用本地状态解决的不上全局
- 服务端状态优先使用项目已有的请求层封装，禁止绕过手写
- 严格依赖 Phase 2 产物，**不引入项目现有方案之外的新库**

## 组件设计原则

- **组合优于继承**：拆成职责明确的子组件组合，不要巨型组件
- **职责单一**：UI 渲染与业务逻辑分离，逻辑写进可测试的独立单元（hook / composable / service）
- **类型安全**：所有组件接口必须有类型定义，禁止 `any` / `unknown` 兜底
- **错误处理**：数据请求子树必须有错误边界或等效容错机制包裹
- **性能预防**：列表超过 100 条考虑虚拟滚动；高开销计算做缓存处理

| UI 模块 | 组件名 | 来源（文件#行号 或 第三方库名） | 结论 | 关联 REQ |
|---|---|---|---|---|

| （示例）提交按钮 | \<ComponentName\> | UI 库原生组件 | 复用 | REQ-002 |

---

## UI 区域与 REQ 映射规则

- UI 改动若影响任一区域，必须同步更新对应 `REQ-*` 的验收项
- 出现「视觉符合但 REQ 未覆盖」的情况，按**未完成**处理
- 出现「REQ 存在但无 UI 对应」的情况，升级为阻塞问题
