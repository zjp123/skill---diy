# Phase 3：UI → 组件映射

## 目标
将 Figma 区域拆解为可编码的组件选型，输出 `ui-analysis-with-components.md`。

---

## 拆解维度

每个 UI 区域必须覆盖以下所有维度：

| 维度 | 内容 |
|---|---|
| 结构 | 布局层级、嵌套关系 |
| 组件选型 | 优先复用 → 小改造 → 自定义（三选一，附代码证据） |
| 状态机 | view / edit / saving / error / empty / disabled |
| 交互流 | 点击路径、数据流向、防抖 / 防重提交策略 |
| 校验规则 | 前端拦截规则 + 后端错误码二次兜底映射 |
| 权限态 | 无权限时的 UI 降级方案（隐藏 / 置灰 / 跳转） |
| 无障碍 | ARIA 角色、键盘导航顺序、焦点管理 |

---

## 状态管理选型决策

在组件映射阶段确定每个数据域的状态归属，避免后续随意混用：

| 数据类型 | 推荐方案 | 判断依据 |
|---|---|---|
| 服务端数据（列表、详情、分页） | React Query / SWR | 需要缓存、自动失效、loading/error 三态 |
| 全局 UI 状态（当前用户、权限、主题） | Zustand / Redux | 多个无关组件共享，生命周期跨页面 |
| 跨步骤表单 / 向导流程 | Zustand 局部 Store 或 React Context | 单一流程内共享，完成后销毁 |
| 单页面表单输入 | React Hook Form | 需要校验、dirty 状态、提交拦截 |
| 组件本地 UI 状态（弹窗开关、编辑态） | `useState` / `useReducer` | 不需要跨组件共享 |

**决策规则**：
- 优先用最小作用域（`useState` 能解决就不用全局 Store）
- 服务端状态禁止用 `useState` + `useEffect` 手写（有 React Query / SWR 就用）
- 确认项目已有哪些方案（Phase 2 产物），**不引入新的状态管理库**

---

## React 组件设计原则

- **组合优于继承**：拆成 Header / Body / Footer 的组合，不要巨型组件
- **职责单一**：UI 渲染组件不写业务逻辑，逻辑写进自定义 hooks
- **Props 类型安全**：所有 Props 必须有 TypeScript 类型定义，禁止 `any`
- **错误边界**：数据请求子树必须有 `<ErrorBoundary>` 包裹
- **性能预防**：列表超过 100 条考虑虚拟滚动；复杂计算用 `useMemo`

## UI 区域与 REQ 映射规则

- UI 改动若影响任一区域，必须同步更新对应 `REQ-*` 的验收项
- 出现「视觉符合但 REQ 未覆盖」的情况，按**未完成**处理
- 出现「REQ 存在但无 UI 对应」的情况，升级为阻塞问题
